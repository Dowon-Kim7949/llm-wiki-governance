# 보안 정책

## 지원 버전

`llm-wiki-governance`는 롤링 릴리스 모델을 따르며, 보안 수정은 npm의
**최신 배포 버전**에 반영됩니다. 신고 전에 최신 버전으로 업그레이드하고,
프로젝트에서는 최근 버전으로 고정해 사용하세요.

| 버전 | 지원 |
|---|---|
| 최신 배포 버전 | ✅ |
| 이전 버전 | ❌ (업그레이드 권장) |

   **Report a vulnerability**. 신고가 비공개로 유지되고, 수정과 보안
   권고(advisory)를 함께 조율할 수 있습니다.

가능한 범위에서 다음을 포함해 주세요:

- 재현 절차(최소한의 명령 시퀀스나 저장소 상태).
- 영향 받는 버전과 환경(OS, Node.js 버전).
- 제안하는 해결 방안.

## 처리 기대치

이 프로젝트는 소규모 팀이 최선 노력(best-effort) 기반으로 관리합니다:

- **접수 확인:** 영업일 기준 약 5일 이내.
- **평가 및 수정:** 진행 상황을 공유하며, 문제가 확인되면 다음 릴리스에서 수정을
  목표로 합니다.
- **공개:** 조율된 공개(coordinated disclosure)를 선호합니다. 수정이 배포되면
  원하는 신고자를 GitHub 보안 권고에 크레딧으로 표기합니다.

## 범위 및 위협 모델

신고 범위를 잡는 데 도움이 되는 사실:

- 이 CLI는 **런타임 서드파티 의존성이 없습니다**(Node.js 내장 모듈만 사용) —
  의존성 공격면이 최소화됩니다.
- **로컬 파일시스템**에서 동작합니다 — 프로젝트 파일을 읽고, 명시적
  `--write`/`--apply` 플래그가 있을 때에만 wiki/adapter 파일을 씁니다. 프로젝트
  내용을 외부로 전송하지 않습니다.
- 리포트에서 의심되는 비밀값을 표시·마스킹하는 **민감정보 스캔** 기능이 있습니다.

특히 관심 있는 신고: 의도 범위를 벗어난 쓰기, 경로 탐색(path traversal), 조작된
프로젝트 파일을 통한 코드 실행, 리포트/로그로의 민감정보 유출.

## MCP 서버 신뢰 모델

`llm-wiki mcp` 서버(Gate 11)는 **로컬·단일 테넌트** 배치를 전제로 설계됐습니다.
노출하기 전에 경계를 이해하세요:

- **전송 전제.** 개행 구분 JSON-RPC 2.0을 **stdio**로 말하며, 신뢰된 클라이언트
  (에디터/에이전트 — Claude Code·Cursor 등 — 또는 CI 러너)가 **로컬 서브프로세스**로
  띄우는 것을 의도합니다. `stdout`이 **프로토콜 채널**이라 프로토콜 메시지만
  나갑니다(로그는 `stderr`). 서버의 stdout에 다른 것을 흘려보내지 마세요.
- **인증·인가 없음.** 서버에는 인증 계층·세션·호출자별 접근 통제가 없습니다.
  프로세스에 닿을 수 있는 누구든 노출된 모든 툴을 호출할 수 있습니다.
- **읽기 전용 툴만.** 쓰기/변경 명령(`init`/`fix`/`migrate`/`drift`/
  `quickstart --write`, 그리고 `review`의 **승격** 경로)은 MCP로 노출하지 않습니다.
  툴은 `readOnlyHint`를 답니다. `verified` 승격은 사람의 CLI 액션으로만 일어나며,
  `review` MCP 툴은 읽기 전용 백로그 **목록**만 노출합니다.
- **네트워크로 노출하지 마세요.** 인증이 없으므로 stdio 서버를 공개/원격 브로커나
  네트워크 전송 뒤에 그대로 두지 **마세요**. 원격 접근이 꼭 필요하면 **직접**
  인증·접근 통제를 갖춘 프록시를 앞에 두고, 그 프록시에 닿는 누구나 위키 전체를
  읽을 수 있다고 간주하세요.
- **민감 저장소.** CLI와 동일한 내용 보호가 적용됩니다: restricted /
  `contains_sensitive_info` / 민감 스캔 히트 문서는 기본적으로
  `list_docs`/`search_docs`에서 제외되고(opt-in `includeSensitive`), 반환 본문/
  스니펫은 민감 라인을 마스킹합니다. 그래도 서버에 닿는 호출자는 제외되지 않은 모든
  문서를 읽을 수 있다고 가정하세요 — 진짜 비밀은 `visibility: restricted`에 의존하지
  말고 아예 위키에 두지 마세요.

프로젝트와 사용자의 안전을 지키는 데 도움 주셔서 감사합니다.
